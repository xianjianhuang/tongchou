# 统筹 — 全技能编排引擎

> 「不需要知道用什么工具，告诉我你要什么结果。」

统筹是 Hermes 技能系统的**大脑**：用户说一个目标，统筹自动判断该用什么技能、按什么顺序执行、怎么组合输出。

---

## 核心理念

**不需要用户知道「有什么技能」**。用户只说目标，统筹自动编排。

### 三位一体架构

| 角色 | 技能 | 职责 |
|------|------|------|
| **女娲（造）** | `huashu-nuwa` | 从0到1创建新技能（人物/主题蒸馏） |
| **达尔文（进化）** | `darwin-skill` | 对已有技能做评估、优化、进化 |
| **统筹（编）** | `tongchou`（本skill） | 分析需求→匹配技能→编排流程→交付 |

## 执行流程

### Phase 0: 需求分析

收到用户输入后，先判断属于哪个类型：

| 用户需求 | 类型 | 对应行动 |
|----------|------|----------|
| 「帮我想个创意方案」 | 创意生成 | → 加载 `brainstorming` + 匹配相关技能 |
| 「分析下这段代码」 | 代码分析 | → 加载 `code-analyzer` |
| 「帮我做个XX的skill」 | 创建skill | → 转交 `huashu-nuwa` |
| 「这个skill质量怎么样」 | 评估skill | → 转交 `darwin-skill` |
| 「帮我整理文件」 | 文件整理 | → 加载 `file-organizer` |
| 「爬这个网站」 | 数据采集 | → 加载 `crawler-stack` |
| 「查一下XX资料」 | 信息调研 | → 加载 `web-research` |
| 「做个图/做个演示」 | 内容创作 | → 匹配对应创意技能 |
| 无法明确分类 | 综合任务 | → 走 Phase 1 技能扫描 |

**核心原则**：尽可能直接命中类型，不要什么都走综合扫描（太慢）。

### Phase 1: 技能扫描（仅综合任务）

当需求无法明确匹配单一类型时，执行技能全景扫描：

1. 调用 `skills_list()` 获取所有可用技能
2. 按以下维度评估每个技能的匹配度：
   - **关键词匹配**：技能描述中是否有用户需求关键词
   - **能力域匹配**：技能属于哪个能力域（创作/开发/运维/数据/调研）
   - **输入输出匹配**：用户提供什么→用户要什么
3. 筛选 Top 3 候选技能，展示给用户选择（或直接按置信度最高的走）

### Phase 2: 编排执行

确定技能组合后，按以下模式执行：

#### 模式 A: 单技能直通
一个技能就能搞定 → 直接加载并执行
- 示例：用户说「查一下黄金价格」→ 加载 `market-quote` → 执行行情查询

#### 模式 B: 串联管线
技能 A 的输出是技能 B 的输入
- 示例：用户说「爬这个网页做个分析报告」
  1. `crawler-stack` → 抓取网页内容
  2. `code-analyzer` → 分析网页结构/数据
  3. 主agent整理报告

#### 模式 C: 并联分流
多个独立子任务并行
- 示例：用户说「帮我看看这几家公司」
  1. 并行：`web-research` × N（每家一个子任务）
  2. 汇总分析

#### 模式 D: 闭环迭代
输出结果后再次评估是否需要进一步操作
- 示例：用户说「这个网站结构不对，帮我改」
  1. 分析问题
  2. 生成修改方案
  3. 用户确认
  4. 执行修改
  5. 验证结果

### Phase 3: 结果交付

- 汇报做了什么技能组合、为什么这么组合
- 展示每个阶段的关键结果
- 问用户：是否满意？需要调整？是否需要保存为新的工作流？

## 触发条件

当用户消息命中以下任意情况时，**必须加载本 skill**：

1. 用户说「安排」「搞定」「一条龙」「帮我搞定」— 明确的编排需求
2. 用户的需求**需要多个技能组合完成**（如「爬这个网站，然后做个分析报告」）
3. 用户的需求**不清楚用什么技能**，属于综合类任务
4. 用户说「有什么技能可以用」「帮我看看怎么做」

**不触发的情况**（直接走对应 skill 即可）：
- 明确的单一技能需求（「画个图」「查股价」「造个芒格的skill」）
- 不需要技能组合的简单对话

## 失败模式与应对

| 失败场景 | 原因 | 应对 |
|----------|------|------|
| 匹配到的 skill 不存在 | skills-router 触发词过期 | 回退：用 skills_list 重新扫描，直接加载匹配的 skill |
| 子 agent（delegate_task）403 欠费 | CodeBuddy/QCLaw 积分耗尽 | 回退：主 agent 直接执行，不走子 agent |
| 用户需求太模糊，无法匹配任何 skill | 缺少领域关键词 | 回退：问用户 1 个关键问题缩小范围，再匹配 |
| 编排的 skill 说「做不到」 | skill 的假设与实际情况不符 | 回退：换备选 skill 或直接用主 agent 能力完成 |
| 串行管线中某个 skill 卡住超时 | 外部依赖不可用（网络/API） | 回退：跳过该 skill，记录失败原因，继续后续步骤 |

## 反例与黑名单

**统筹不做的事：**
- ❌ 不强行套用不匹配的 skill — 没有合适 skill 时用自己的能力完成
- ❌ 不给用户列「你有这些技能可用」清单 — 那是 skills_list 做的事，统筹直接干
- ❌ 不反复确认 — 高置信度直接执行，低置信度问 1 次就定方向
- ❌ 不把简单需求复杂化 — 单 skill 能搞定的事不走编排
- ❌ 不改 skill 内容 — 那是达尔文的工作，统筹只编排不修改

## 检查点

**Phase 0 完成时** 🔴 STOP — 展示匹配结果让用户确认
```
用户需求 → 匹配到 [skill A, skill B, skill C]
执行模式：串行/并行/混合
确认继续？
```

**Phase 2 完成时** 🔴 STOP — 展示执行结果
```
已完成：...
未完成：...（原因）
是否需要调整？
```

## 与 skills-router 的关系

- **skills-router**：简单匹配 → 用户说了触发词就加载对应 skill
- **统筹**：智能编排 → 用户没说触发词但需要组合技能时，分析后编排

两者互补。skills-router 负责「用户知道要什么」的场景，统筹负责「用户不知道要什么/需要组合」的场景。
